# Managing Credentials

Workflows that call external APIs need authentication tokens. liteflow provides encrypted credential storage, automatic auth header injection into HTTP steps, and credential testing -- so you never hardcode secrets into workflow definitions.

---

## Why Credentials Matter

Any workflow that interacts with an external service (GitHub, Slack, OpenAI, etc.) needs an API token. Passing tokens directly in step configs is insecure and fragile. liteflow's credential system solves this by providing:

- **Encrypted storage** -- Tokens are never stored in plaintext. They are encrypted with Fernet symmetric encryption (or base64 XOR obfuscation as a fallback).
- **Auto-injection** -- HTTP steps that reference a service name instead of a full URL automatically receive the correct auth headers.
- **Service-aware formatting** -- Each service gets its own auth header format (e.g., GitHub uses `token`, Anthropic uses `x-api-key`).
- **Credential testing** -- Verify that a stored token actually works before running a workflow.

---

## Adding a Credential

To store a credential for a service, use the `flow-auth` command:

```
/liteflow:flow-auth github
```

Claude will prompt you for the API token. Once provided, the token is encrypted and stored in `~/.liteflow/credentials.db`.

**How encryption works:**

1. A machine-derived key is generated by taking the SHA-256 hash of `platform.node()` (your hostname).
2. If the `cryptography` package is installed, the token is encrypted using **Fernet symmetric encryption** with this key.
3. If `cryptography` is not available, the token is obfuscated using **base64 XOR** with the derived key. A warning is issued in this case.

## Supported Services

liteflow ships with built-in knowledge of these services:

| Service | Base URL | Auth Header Format |
|---------|----------|-------------------|
| github | `https://api.github.com` | `Authorization: token {token}`, `Accept: application/vnd.github.v3+json` |
| slack | `https://slack.com/api` | `Authorization: Bearer {token}` |
| jira | `https://your-domain.atlassian.net/rest/api/3` | `Authorization: Bearer {token}` |
| notion | `https://api.notion.com/v1` | `Authorization: Bearer {token}` |
| linear | `https://api.linear.app/graphql` | `Authorization: Bearer {token}` |
| sendgrid | `https://api.sendgrid.com/v3` | `Authorization: Bearer {token}` |
| openai | `https://api.openai.com/v1` | `Authorization: Bearer {token}` |
| anthropic | `https://api.anthropic.com/v1` | `x-api-key: {token}`, `anthropic-version: 2023-06-01` |

Custom services can also be stored. Any service name not in the table above uses `Authorization: Bearer {token}` by default.

## Testing Credentials

Before running a workflow, verify that a stored token is valid:

```
/liteflow:flow-auth test github
```

This performs a lightweight API call to check authentication:

| Service | Test Endpoint | Method |
|---------|--------------|--------|
| github | `/user` | GET |
| slack | `/auth.test` | GET |
| openai | `/models` | GET |
| anthropic | `/models` | GET |

The command returns a result like `{"valid": true, "message": "Successfully authenticated with github"}` or `{"valid": false, "message": "Authentication failed (401): Unauthorized"}`.

For services without a known test endpoint, liteflow confirms the token exists but notes that it cannot verify it against an API.

## Listing and Removing

**List all stored credentials** (tokens are always masked in output):

```
/liteflow:flow-auth list
```

**Remove a credential:**

```
/liteflow:flow-auth remove slack
```

You will be asked to confirm before the credential is deleted.

## How Credentials Flow into Workflows

When an HTTP step references a service name instead of a full URL, the credential system activates automatically. Here is what happens at execution time:

1. **URL resolution** -- `HTTPStep._resolve_url()` checks whether the `url` field starts with `http://` or `https://`. If it does not, it treats it as a service name and looks up the base URL -- first from the credential's stored `base_url`, then from the built-in `SERVICE_URLS` dictionary.
2. **Auth injection** -- `HTTPStep._inject_auth()` retrieves the decrypted token from `SecureStore` and adds the appropriate auth header for that service.
3. **Skipped when unnecessary** -- Auth injection is skipped if the URL is already a full `http://` or `https://` URL, or if an `Authorization` header is already present in the step config.

**Example HTTP step config:**

```json
{
  "type": "http",
  "url": "github",
  "endpoint": "/repos/{repo}/pulls",
  "method": "GET"
}
```

At runtime, the engine:

1. Resolves `"github"` to `https://api.github.com`
2. Appends the endpoint: `https://api.github.com/repos/{repo}/pulls`
3. Substitutes `{repo}` from the current context
4. Injects `Authorization: token <decrypted_token>` and `Accept: application/vnd.github.v3+json`

If you need to call a URL directly and bypass credential injection, use a full URL:

```json
{
  "type": "http",
  "url": "https://httpbin.org/get",
  "method": "GET"
}
```

## Security Model

- **Encryption** -- Tokens are encrypted with Fernet symmetric encryption using a key derived from `platform.node()` (your machine's hostname). The raw SHA-256 hash is base64-encoded to produce the Fernet key.
- **Fallback** -- If the `cryptography` package is not installed, tokens are obfuscated with base64 XOR using the same derived key. A warning is issued at runtime. Install `cryptography` for proper encryption: `pip install cryptography`.
- **Masked output** -- Credential values are never displayed in command output. The `flow-auth list` command shows service names and metadata only.
- **Storage location** -- All credentials live in `~/.liteflow/credentials.db`, a `sqlitedict`-backed SQLite database.
- **Machine-bound** -- Because the encryption key is derived from the hostname, credentials are not portable between machines. Moving `credentials.db` to a different machine will fail to decrypt.

## Credential Metadata

When storing credentials programmatically via the `SecureStore` API, you can attach metadata such as scopes and descriptions:

```python
from lib.creds import SecureStore

store = SecureStore()
store.set_token(
    "github",
    "ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
    metadata={
        "scopes": ["repo", "user"],
        "description": "Personal access token for CI workflows",
    },
)
```

Metadata is stored alongside the encrypted token in the credential record, together with an `added_at` UTC timestamp. This metadata can be useful for auditing which tokens are stored and what permissions they carry.
